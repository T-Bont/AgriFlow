1. Executive Summary
AgriFlow is a simplified, offline-first farm management PWA designed to replace paper notebooks for family farms. It focuses on geospatial navigation (fields on a map) and high-speed financial/agronomic data entry. The core technical differentiator is its ability to handle complex field-level P&L calculations while functioning entirely without cellular service.
2. User Personas
Persona A: "Harvest Harry" (Primary)
- Profile: 58-year-old owner/operator. Uses a mid-range Android phone. Eyesight is average; hands are often gloved or dirty.
- Tech Literacy: Low. Gets frustrated by small buttons, loading spinners, and login screens.
- Behavior: Checks the app while in the tractor cab or at the gas station.
- Key Constraint: "If I have to tap more than 3 times to enter a fuel receipt, I’m going back to my notebook."
3. Functional Requirements
3.1 Geospatial Dashboard (Home)
- Map View: 
	- Implement Mapbox GL JS.
	- Render saved fields as Polygons overlaying the satellite layer.
 	- Color Coding: Fields color-coded by crop type (e.g., Corn = Yellow, Soy = Green) or Profitability (Green = Positive, Red = Negative).
 	- Interaction: Tapping a polygon opens the Field Detail View.
Field Creation:
- "Draw on Map" feature to define field boundaries (GeoJSON).
- Auto-calculate "Acres" based on polygon area, but acreage can also be manually entered if they have the information.
3.3 Field-Level Financials & Data Entry
- The "Log" Action: A Floating Action Button (FAB) (+) on the dashboard and field view to add a record.
- Transaction Types:
1. Income: Subsidies, Insurance, Commodity Sales.
2. Expense: Inputs, Machine Hire, Taxes, Interest.
3.4 Data Visualization
- Charts: Use Recharts for lightweight mobile rendering.
- Views:
	- Field View: Stacked Bar Chart (Expenses broken down by category vs. Total Income).
	- Global View: Net Income per Acre (Leaderboard of most profitable fields).
3.5 Farm Configuration & Entity Logic (The "Sharecropping" Constraint)
- Problem: The user may not own 100% of every crop.
- Requirement:
- Season Setup: When starting a new season (e.g., "2025 Corn") for a specific field, the user must define:
- Operator Share %: (e.g., 100% for owned land, 66% for sharecropped).
- Landlord Name: (Optional text field).
- Impact: All "Net Income" calculations on the dashboard must apply this percentage multiplier to Revenue (and potentially shared expenses).
3.6 Advanced Financial Recording
A. Revenue Streams (Detailed)
1. Quality Metrics (The "Dockage" Check):
- Input fields for Moisture %, Test Weight, and Protein.
- Why? High moisture reduces net weight/pay. This data is critical for the "Per Acre" audit.
2. Government Payments:
- Mus be able record not just that there were government payments but what kind of payments they were.
- Program Selector: Must allow selecting specific subsidy programs (eg. PLC, ARC, CFAP, ERP, MFP) this should be a dropdown field, but they should have the ability to add more to it when new programs arise.
- Why? Farmers need to know if they are profitable from farming vs. subsidies.
B. Expense Recording (Vendor & Product Context)
1.	Vendor Tracking:
- Every expense must have a "Payee/Vendor" field (e.g., "MKC", "CPS", "Sprayer Co").
- UI Requirement: Auto-complete based on previous entries.
2.	Chemical/Fertilizer Specifics:
- Product Name: Instead of just "Herbicide," have the ability to capture specific names like "Liberty 280", "Zidua", "Finesse".
- Application Stage: Dropdown to indicate timing (at what stage of crop growth was it applied):
- Options: Pre-Plant, Starter, Top Dress, Spring Weed, Fall Weed, Desiccant.
- Weather Context: A generic weather_notes field (e.g., "Applied 12hrs before rain").
3.7 Nutrient Loading (The "Actual N" Calculation)
- Requirement:
- When logging fertilizer, providing a "Calculator" helper:
- Input: Total Lbs of Product (e.g., Urea 46-0-0).
- Input: Product N-P-K rating (e.g., 46-0-0).
- Output: Auto-calculate Actual Lbs of N per Acre.
- Data Storage: Store both the Cost and the Actual N value.
4. API & Sync Strategy
4.1 Technology Choice: TanStack Query + Supabase
We will not build a custom REST API. We will use the Supabase JavaScript Client directly in the React frontend, wrapped in TanStack Query for caching and state management.
3. Commodity API (3rd Party):
- Use a serverless Edge Function (Supabase Functions) to proxy requests to an API like Alpha Vantage or Commodities-API.
- Why? To hide API keys and cache results (reduce 3rd party costs).
5. UI/UX & Design Constraints
- Input Types:
- Money fields must default to the numeric keypad.
- Date fields must use the native OS date picker.
- Feedback:
- Save actions must show a "Toast" notification: "Saved to device" (Offline) or "Synced to Cloud" (Online).
- To prevent overwhelming the user with these new fields, we will use Conditional Logic in the forms (however for debugging and testing purposes in the initial phase of the project there needs to be a toggle that can enable or disable conditional logic in the forms thus allowing the developer to see all the forms at once without clicking through them).
- Default State: The "Add Record" screen shows only Amount, Date, and Category.
- Condition 1 (Fertilizer): If Category = "Fertilizer", slide down inputs for Vendor, Product Name, N-P-K Calculator, Application Stage.
- Condition 2 (Grain Sale): If Category = "Grain Sale", slide down inputs for Delivery Location, Contract ID, Moisture/Test Weight.
- Condition 3 (Gov Payment): If Category = "Gov Payment", slide down Program Type selector.
6. Feature: The Commodity "Action" Dashboard
Objective: To give the user an overview of the current market conditions for their crops and give them more specific financial market related information.
A. The Tickers
- Visual: Rows of crop prices specific to which crops the user has. This would have typical information associated with each one like current price, % change, etc.
- Content:
- Buhler Tickers: ZC (Corn), ZS (Soybeans), KE (Wheat), Milo same as corn.
- Market: Chicago Board of Trade (CBOT) Futures for Corn (ZC), Soybeans (ZS), and Wheat (ZW) and others.
- Display: Big bold price (e.g., "$4.50"), Change indicator (Green Up Arrow / Red Down Arrow), and " cents change" (e.g., "+0.12").
- User Config: User selects which crops appear here in Settings.
B. The "Local Basis" Widget (The Critical Differentiator)
- Problem: Farmers don't sell at the Board price; they sell at the Cash Price (Board + Local Basis).
- Functionality:
- Location Tracking: The user can "Follow" specific local elevators (e.g., "ADM South", "Buhler Co-op").
- API Integration: Fetch Cash Bids for these specific locations (via 3rd party API like Barchart or DTN).
- Visualization: A simple leaderboard card:
Corn Cash Bids
- ADM South: $4.35 (Basis: -0.15)
- Buhler: $4.28 (Basis: -0.22)
- Ethanol Plant: $4.45 (Basis: -0.05) Highlighted as Best Price
C. The "Profit Overlay" Chart (The Killer Feature)
- Concept: This charts the Market Price against the User's Real Data.
- X-Axis: Time (Next 6 months).
- Y-Axis: Price per Bushel.
- Line 1 (Market): The projected Futures price curve.
- Line 2 (The "Red Line"): The user's Break-Even Price.
- Calculation: (Total Field Expenses / Estimated Yield).
- Dynamic: As the user enters more expenses (fertilizer, fuel) or lowers yield estimates, this red line moves up in real-time.
- Fixed: After the harvest the user will enter the actual amount of yield, so now the breakeven line can remain at the price position where the field will breakeven (ie. If the user has 1000 bushels of X, and it costed them $1000 overall, the breakeven indicator would be at $1/bushel)
- User Value: Harry can look at the chart and instantly see: "Corn is $4.50, but my breakeven is $4.60. I cannot sell today."
D. Marketing Position Summary (Inventory Check)
- Purpose: To track how much grain is "Unsold" vs. "Contracted."
- Data Source: Aggregated from transactions (Harvest logs vs. Sales logs).
- Visual: A "Battery" style progress bar for each crop.
- Total Inventory: 10,000 Bushels (Harvested).
- Sold/Forward Contracted: 4,000 Bushels (40%).
- Unsold (Risk): 6,000 Bushels (60%).
- Action Button: "Add Contract" button next to the Unsold amount to quickly log a forward sale.
7. Inventory Visualizations
Objective for AI Agent: Do not build generic charts. The visualizations in this app must merge the user's specific breakeven data (view_field_pnl) with their inventory data (contracts and transactions). The goal is to show the user their "Risk" (unsold grain) and their "Opportunity" (current market price vs. breakeven).
Visual Component 1: The "Cross-Season" Inventory Battery
Location: Inventory Page (Independent of the Global Crop Year Toggle).
Purpose: To show the farmer exactly how much grain they have to sell, explicitly separated by Crop Year so old inventory isn't mixed with new projections.
UI Layout: A stacked horizontal progress bar (Battery style) for each active crop/year combination (e.g., one bar for "2025 Corn", one for "2026 Corn", the bars should be grouped by crop so it should go "2025 Corn" thne below that "2026 Corn" then below that it goes to the next crop so "2025 Wheat").
Data & Logic Requirements:
Grouping: Query the database and group by season.year and season.crop_type.
The "Total" (Width of the bar): * If Pre-Harvest: Eventually this will be an actual prediction of yield, but for now just deafult to a predicted yield of 1000 bushles for the field for that year.
If Post-Harvest: Sum of the actual amount of bushels harvested for that field in the given year.
The "Sold" Segment (Green color): * Sum of quantity_bushels sold.
The "Unsold/Risk" Segment (Orange/Yellow color): * Total - Sold.
